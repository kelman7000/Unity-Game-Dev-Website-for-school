Layers

In this section you will learn about layers, both the sorting layers used in 2D games and regular layers for all games.

Sorting Layers

Sorting layers are used with Sprites in a 2D game and they are for sorting the overlay order of the sprites. When the Sprite renderer component is added to a gameObject then under Additional Settings you will see that there is the Sorting Layer dropdown. There is also an order in layer so you can overlap sprites on the same sorting layer any way you want.

To add a new Sorting Layer all you need to do in click Add Sorting Layer... at the bottom of the dropdown. Or Edit->Project Settings-> tags and layers.

Layers

In the inspector window to the right of the Tag dropdown you may have noticed the Layer dropdown as well. These layers are usful for many things but here I'll just talk about how they help us when working with physics.

Layers are helpful when we are working with physics because if we go to Edit -> Project Settings in the top left hand corner then go to physics 2D and scroll down you will see the Layer Collision Matrix dropdown. Using this we can decide what objects can collide with and what they will pass right through. You can add a new layer the same way that a sorting layer is added.

Colliders

There are a veriety of colliders in Unity, when you click add component and search for collider you will notice that there are 2D and 3D colliders. Because we are focusing on 2D games we can ignore all things 3D.

List of 2D Collisers

  • Circel Collider 2D
  • Box Collider 2D
  • Polygon Collider 2D
  • Edge Collider 2D
  • Capsule Colider 2D
  • Composite Collider 2D

While there are six different 2D colliders they all function the same essentialy. When you add one to an object there will be a light green outline, if it does not fit the size of the object to your liking you can easly change it with the Edit Collider button. Just click that and you will be able to change the collider around. The only other part we will discuss here is Is Trigger this will make it so other object don't collide with it but pass through. Below I will link to some resources that will help you gain a better understanding.

  • Unity Manual Collider 2D
  • Scripting API Collider 2D - read to Messages

Colliders work hand in hand with the Rigidbody2D without colliders objects would just pass right through each other. Now as you may recall from the layers section we have the ability to change what objects can collide with.

Rigidbody

The Rigidbody is what allows a GameObject to be effected by gravity and the rest of the physics engine behind Unity. You can use the Rigidbody within other scripts that you write to manipulate the gameObject with the attached rigidbody. The rigidbody has three different Body Types (Dynamic, Kinematic, and Static). The Dynamic body type will make the object be effected by all forces, the Kinematic body type requires you to apply some force to the object in order for it to move. I won't discuss the Static body type because it is not used here. To learn more visit Unity Manual Rigidbody2D

Prefabs

When making a game you will need to re-use game assets, these assets are called Prefabs. Prefabs are copies of a game object, take for example an enemy. You don't want to create a new enemy from scratch every time you need one, thats why you would make a prefab. It will store all the data about that enemy for easy re-uses. They are also very usful if we are spawning them into the game scene randomly, as without the prefab how would be tell the script what to spawn in.

NOTE: this is done using instantiate which is shown in the projects here

To create a Prefab first make a new folder called Prefabs then drag and drop the gameObject you want to make a prefab out of into that folder. Now you have a prefab, you can drag that into the scene as many times as you want and making a change to the prefab will update all the copies that are in the game hierarchy. To learn more visit Unity Prefabs - The Complete Animated Guide | Game Dev Classroom

Physics2D

When making games there is usually going to be some kind of physics interactions taking place, this could be collisions or even just the player jumping. Either way having a good understanding of the Unitys 2D physics engine will help you develop good 2D games.

Getting Started

If you go to Edit -> Projects Settings -> Physics 2D you will open a window that allows you to limit the accuracy of physics simulations. One thing you should always keep in mind, is that the higher the accuracy the more time spend processing. When starting out in Unity you will most likely not need to do anything in this window other than interact with the Layer Collision Matix.

This matrix allows us to pick and choose what layers can collide with each other. If you look at the check box in the far left column at the bottom row you can see it is checked this means that the objects in this Obsticals can collide with one another, where as the objects in the Bullet layer cannot collide with each other.

Scripting API

Now you may not need to change any of the settings but you will be using physics in your scripts. For example you may need to apply an upward force to the player object to make them jump and come backdown, but in order for them to jump you have to check that they are touching the ground. There is lots to cover here so we will only be covering a small portion of it, mainly what you will need to no for basic player controllers. Some of what we cover will be directly related to the rigidbody2D.

Distance

This method calculates the minimum distance between two colliders. It could be used if you want the player to be within x units of some enemy before it starts to move closer to them.

IsTouchingLayers

This method will check weather the collider is touching any other colliders on some specific layerMask or not. It can be used to see if the player is touching the gound.

Events

For this section we are goin to go over how events work and what they help us do. You may find your self asking why events are being covered here and not in the C# section of the site. Well the reason is that the scripting side of things is not nearly as important as how to use they effectivly in Unity. To see events in action please go to Top-Down shoot in the projects.

Example: Were to use an event.

Lets say that in our game we have a GameController, which keeps track of how many deaths the player has. One way we could go about this would be to get a reference to the GameController in the Player scrip, and in the Die() method of Player.cs add some logic for the GameController. There are many issues with this approach though, one of which is that the Player script now has a dependancy on the GameController. Another is wee may need to have the same logic we just wrote execute if something else happens to the player and now we need to write the logic we already have. We're programmers, and that means we are lazy and don't like doing the same thing over and over again. Now on the other hand if we are to use an event to handle the players death all we would need to do is create some event playerDeathEvent and we invoke it in the players death method. Then in the GameController script we subscribe some other method to the playerDeathEvent and all the logic in that method will be executed when the player dies. But we can now subsribe this method to other events and we only had the write the logic once.

Take Away

What you should take away from this is that events can make our code cleaner and DRY. If you want to know more I suggest you watch [C#] Delegates and Events (E01: delegates) by Sebastian Lague and also [C#] Delgates and Events (E02: events) by Sebastian Lague. Both of these will help you under stand events much better.

Staying Organized

Here I will just be telling you about some ways that you can keep everything organized in both the editor and in Visual Studio.

In The Editor

Basicaly the only thing I can tell you here is to make folder and keep things grouped together with those. To make a new folder just right click in the project window and go to create.

Visual Studio

There many ways to stay organized in visual studio, you can ground methods and fields by public, private an proteced and much more. To keep the fields that are visible in the inspector use [Header("something")] to group fields or [Space()] to put a gap between them. But it is also just as important to keep the code it self clean and easy to read. use #region NAME #endregion to group methods and fields to gether.

As I have not shown [SerializeField] you propably do not know what it does. Some times there are variables that we want to access in the inspector and the are private so we use that to make them visible there, if they are just public then they will show there without it.
